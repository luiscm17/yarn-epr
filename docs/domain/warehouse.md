## 1. Bounded Context

Warehouse manages inventory (MP, PT, Supplies) and lot identity. It shares the
**Lot** entity with Operation, but each context owns different data:

```
┌─────────────────────────────────┐
│          WAREHOUSE                │
│                                   │
│  Lot (identity owner)             │
│    ├── code, title, color, client │
│    ├── specifications             │
│    └── movements (append-only)    │
│                                   │
│  Movement (per subdomain)         │
│  StockBalance (calculated)        │
│  Supplier                         │
│  Client                           │
│  SuppliesCategory                 │
│  SuppliesItem                     │
└──────────┬──────────────────────┘
           │ code
           ▼
┌─────────────────────────────────┐
│          OPERATION                │
│                                   │
│  Lot (reads identity only)        │
│    ├── stage records              │
│    ├── observations               │
│    └── quality classification     │
└─────────────────────────────────┘
```

**Shared catalogs** (owned by Warehouse or shared):
- Supplier — Warehouse
- Client — Warehouse
- Title — shared
- Employee — system-wide

## 2. Core Entities

### 2.1 Lot (Warehouse perspective)

The same lot that flows through Operation. Warehouse owns its identity and
creates it during MP reception. The lot crosses contexts via its code.

| Attribute | Description | Notes |
|---|---|---|
| `code` | Unique identifier `NN-GGGG-NNN` | Generated by the system |
| `title` | Yarn thickness designation | Set during reception or enrichment |
| `color` | Requested color | Set during enrichment |
| `client` | Destination client | Set during enrichment |
| `supplier` | Original MP supplier | From reception data |
| `invoice` | Supplier invoice number | From reception |
| `status` | Current warehouse state | See lifecycle below |
| `totalKgReceived` | Total gross weight received | Sum of reception movements |
| `totalKgEmitted` | Total kg emitted to Operation | Sum of emission movements |
| `currentBalanceKg` | Calculated: received - emitted | Stock formula |
| `qualityClassification` | Final quality (read from Operation) | Read-only reference |

**Lifecycle stages (Warehouse view):**

```
Received → Enriched → Emitted → (in Operation) → PT Received → Stored → Sold / Transferred
```

---

### 2.2 Movement

The central record in Warehouse. Every entry or exit is an immutable movement.
This is the core pattern — append-only, tracked, never modified.

| Attribute | Description |
|---|---|
| `movementNumber` | Unique, correlative per type and year |
| `date` | When the movement occurred |
| `subdomain` | Which subdomain: `RawMaterial`, `FinishedProduct`, `Supplies` |
| `movementType` | Specific type within the subdomain (see §5) |
| `direction` | `Entry` or `Exit` |
| `lot` | Lot code (for MP and PT) |
| `quantity` | Amount in the subdomain's unit of measure |
| `unitOfMeasure` | kg, L, pieces, etc. |
| `authorizedBy` | Person who authorized the movement |
| `responsible` | Person who executed the movement |
| `concept` | Brief description |
| `referenceMovement` | Optional — for corrective or return movements |
| `isClosed` | Whether this movement is in a closed period |

**Rules:**
- Movements are append-only — never deleted or edited
- A correction requires a new movement referencing the original
- Once a period is closed, its movements become read-only

---

### 2.3 StockBalance

Not a stored entity — a **calculation** derived from movements.

```
CurrentBalance = PreviousBalance + Σ(Entry quantities) − Σ(Exit quantities)
```

Applies to:
- **MP/PT:** Per lot code, in kg
- **Supplies:** Per item, in the item's unit of measure

---

### 2.4 Supplier

| Attribute | Description |
|---|---|
| `name` | Supplier name |
| `code` | Unique identifier |
| `contactInfo` | Phone, address |
| `isActive` | Whether actively used |

Used by: Raw Material (reception), Supplies (reception).

### 2.5 Client

| Attribute | Description |
|---|---|
| `name` | Client name |
| `code` | Unique identifier |
| `type` | `Direct` (full lot) or `Comercializacion` (units) |

Used by: Finished Product (sales, transfers), Raw Material (enrichment).

## 3. Subdomain Entities

### 3.1 Raw Material

#### 3.1.1 RawMaterialReception

A movement of type `Reception`. Records the arrival of MP bales.

| Attribute | Description |
|---|---|
| `movement` | Inherits Movement attributes |
| `supplier` | From Supplier catalog |
| `invoiceNumber` | Supplier invoice/receipt |
| `grossWeightKg` | Gross weight received |
| `title` | Yarn title |
| `colorOrFiber` | Fiber type or color |
| `truckNumber` | Distributor truck identifier |

**After reception:** The system auto-generates the lot code `NN-GGGG-NNN`
where NN = truck number, GGGG = management year, NNN = sequential.

#### 3.1.2 LotEnrichment

Not a separate movement. Adds order data to an existing lot before emission.

| Attribute | Description |
|---|---|
| `lot` | Existing lot to enrich |
| `client` | Order destination |
| `color` | Requested color |
| `title` | Yarn title (confirmed) |
| `typeN_CH` | Variant (N fiber, fine CH) |
| `observations` | Order notes |

#### 3.1.3 EmissionToProduction

A movement of type `Emission to Production`. Sends MP from Warehouse to Operation.

| Attribute | Description |
|---|---|
| `movement` | Inherits Movement attributes |
| `receivingSupervisor` | Shift supervisor receiving the MP |
| `destination` | Production section |
| `emittedKg` | Quantity emitted |

---

### 3.2 Finished Product

#### 3.2.1 PTReception

A movement of type `Reception from Production`. Records the lot returning from
Operation as finished product.

| Attribute | Description |
|---|---|
| `movement` | Inherits Movement attributes |
| `lotCode` | The original NN-GGGG-NNN |
| `description` | Full item description |
| `receivedKg` | Weight received |
| `value` | Monetary value |
| `category` | From PT catalog |
| `subcategory` | From PT catalog |
| `storageState` | `Bagged` or `Bulk` |

**Validation:** Warehouse verifies the lot data matches Operation's records.
Discrepancies reject the reception.

#### 3.2.2 PTStorageState

The physical state of the PT in the warehouse.

| Attribute | Description |
|---|---|
| `lot` | Lot code |
| `state` | `Bagged` or `Bulk` |
| `changedAt` | When the state changed |
| `changedBy` | Who performed the change |

#### 3.2.3 DirectSale

A movement of type `Direct sale`. Exit of a full lot to a client.

| Attribute | Description |
|---|---|
| `movement` | Inherits Movement attributes |
| `client` | Buying client |
| `invoiceNumber` | From invoice booklet |
| `soldKg` | Quantity sold |
| `authorizedBy` | Jefe de Producción |

#### 3.2.4 ComercializacionTransfer

A movement of type `Comercializacion transfer`. Exit of units to the
commercialization area.

| Attribute | Description |
|---|---|
| `movement` | Inherits Movement attributes |
| `destination` | `Comercializacion` |
| `units` | Quantity in units (not kg) |

#### 3.2.5 Return

A movement of type `Return`. Entry of returned product.

| Attribute | Description |
|---|---|
| `movement` | Inherits Movement attributes |
| `originalSale` | Reference to the original sale movement |
| `returnType` | `Total` or `Partial` |
| `returnedKg` | Quantity returned |

---

### 3.3 Supplies

#### 3.3.1 SuppliesCategory

A growable, user-defined category for supplies.

| Attribute | Description |
|---|---|
| `name` | Category name |
| `description` | Optional description |
| `defaultUnit` | Default unit of measure for items in this category |

#### 3.3.2 SuppliesItem

A specific supply product within a category.

| Attribute | Description |
|---|---|
| `category` | SuppliesCategory |
| `name` | Item name |
| `unitOfMeasure` | Its own unit (may differ from category default) |
| `currentStock` | Calculated from movements |

#### 3.3.3 SuppliesEntry

A movement of type `Reception from supplier`.

| Attribute | Description |
|---|---|
| `movement` | Inherits Movement attributes |
| `item` | SuppliesItem |
| `supplier` | Supplier |
| `quantity` | Quantity in item's unit |
| `entryType` | `Storage Entry`, `Return`, `Sample` |

#### 3.3.4 SuppliesExit

A movement of type `Production consumption` or `Return to supplier`.

| Attribute | Description |
|---|---|
| `movement` | Inherits Movement attributes |
| `item` | SuppliesItem |
| `destination` | Production area |
| `authorizedBy` | Shift Supervisor |
| `quantity` | Quantity in item's unit |

## 4. Movement Types Summary

| Subdomain | Type | Direction | Authorization |
|---|---|---|---|
| Raw Material | Reception | Entry | — |
| Raw Material | Emission to Production | Exit | Jefe de Producción |
| Finished Product | Reception from Production | Entry | — |
| Finished Product | Direct sale | Exit | Jefe de Producción |
| Finished Product | Comercialización transfer | Exit | Jefe de Producción |
| Finished Product | Return | Entry | — |
| Supplies | Reception from supplier | Entry | — |
| Supplies | Production consumption | Exit | Supervisor de turno |
| Supplies | Return to supplier | Exit | — |

## 5. Key Business Rules

1. **Append-only movements:** All movements are immutable. No edits, no deletes.
   Corrections require new movements referencing the original.
2. **Lot identity is unique:** The code `NN-GGGG-NNN` is the single source of truth
   linking MP → Operation → PT. No lot is orphaned.
3. **Stock formula:** `(Previous Balance + Entries) − Exits = Balance`. Calculated
   on demand, always from movement history.
4. **Period closure:** Closed periods set movements to read-only. New movements
   go to the current open period.
5. **Reception verification:** PT is only accepted if its data matches Operation's
   records. Discrepancies are rejected back to Operation.
6. **Authorization by role:** Emissions and sales require Jefe de Producción.
   Supplies consumption requires Shift Supervisor.
